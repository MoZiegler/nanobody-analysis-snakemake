# Nanobody Pipeline

A reproducible, modular, and extensible Snakemake workflow for nanobody (VHH) sequence analysis, structure prediction, and developability scoring.

## Features

- Snakemake pipeline orchestration from raw FASTA to ranked report.
- Optional multiple sequence alignment (MSA) output for downstream inspection.
- Modular structure prediction backend system (Strategy + Factory).
- Optional HPC-friendly execution (`--cluster ...`) and CLI config overrides.
- Reproducible local lightweight setup with `venv`, plus optional Docker and devcontainer.
- Optional GROMACS placeholder simulation (Docker-only mode) that integrates into scoring.

## Setup

### Local (venv + pip)

This mode is the lightweight pipeline runtime. It supports preprocessing, feature extraction,
structure backends, optional MSA export, and scoring, but does not require or run GROMACS.

```bash
python3.11 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

On Windows PowerShell, use:

```powershell
py -3.11 -m venv venv
.\venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

### Optional Docker

Use Docker when you want the optional GROMACS step (`use_gromacs=true`).

```bash
docker build -f docker/Dockerfile -t nanobody-pipeline .
docker run --rm nanobody-pipeline
```

### Optional VS Code Devcontainer

Open the folder in VS Code and choose **Reopen in Container**.
The devcontainer is built from `docker/Dockerfile`, so it uses the same container stack (including GROMACS) as Docker runs.

## Usage

### Lightweight local run (venv)

Run the complete pipeline (mock structure prediction):

```bash
snakemake --cores 4
```

Override structure backend:

```bash
snakemake --cores 4 --config structure_backend=esmfold
```

Suppress MSA generation (default is enabled):

```bash
snakemake --cores 4 --config run_msa=false
```

### GROMACS-enabled run (Docker)

GROMACS mode is intended for Docker images that include system GROMACS.

Linux/macOS shell:

```bash
docker run --rm -v $(pwd):/app -w /app nanobody-pipeline snakemake --cores 4 --config use_gromacs=true
```

Windows PowerShell:

```powershell
docker run --rm -v ${PWD}:/app -w /app nanobody-pipeline snakemake --cores 4 --config use_gromacs=true
```

HPC-style cluster submission example:

```bash
snakemake --cluster "sbatch -A project -t 01:00:00"
```

## Architecture

Pipeline steps:

1. `preprocess`: clean FASTA sequences (`data/input.fasta` -> `data/cleaned.fasta`)
2. `msa` (optional): progressive multiple sequence alignment (`data/msa.fasta`)
3. `msa_summary` (optional): per-sequence MSA statistics (`data/msa_summary.csv`)
4. `features`: compute physicochemical features (`data/features.csv`)
5. `structure`: predict structures with selected backend (`data/structures.json`)
6. `gromacs_simulation` (optional): placeholder MD refinement (`data/md_results.json`)
7. `scoring`: merge features + structures (+ optional MD) and generate ranked report (`results/report.csv`)

Backend system (`scripts/structure.py`):

- `StructurePredictor` interface defines the strategy contract.
- Factory function `get_predictor(name)` resolves backend implementation.
- Backends: `mock`, `esmfold` (real ESMFold inference via ESM Atlas API), `alphafold3` (configured API), `rosetta` (configured API).
- Failures trigger a logged fallback to `mock` so the workflow remains robust.

## Outputs

- `results/report.csv`: ranked developability report
- `data/msa.fasta`: optional aligned FASTA generated by the MSA step (`run_msa=true`)
- `data/msa_summary.csv`: optional per-sequence MSA quality summary (`run_msa=true`)
- `data/pdb/*.pdb`: per-sequence structure files saved after structure prediction when PDB payload is available (for example, `esmfold`)
- `logs/*.log`: per-rule logs with runtime and backend/fallback information

### `data/msa.fasta` description

- Contains an aligned FASTA representation of all cleaned sequences.
- Generated by a progressive, reference-guided MSA procedure in `scripts/msa.py`.
- Alignment uses Biopython pairwise global alignment to iteratively merge each sequence
	into an expanding alignment.
- This is a lightweight pipeline-friendly alignment implementation and is useful for
	inspection/comparison, but should not be treated as a replacement for dedicated
	production MSA tools (for example MAFFT/Clustal/Omega).

### `data/msa_summary.csv` description

Generated from `data/msa.fasta` by `scripts/msa_summary.py`.

- `sequence_id`: FASTA identifier.
- `alignment_length`: Total alignment width (including gaps).
- `nongap_length`: Number of non-gap residues in the aligned sequence.
- `gap_count`: Number of `-` gap characters.
- `gap_fraction`: `gap_count / alignment_length`.
- `consensus_identity`: Fraction of non-gap positions that match a simple per-column
  consensus residue.

Interpretation:

- Lower `gap_fraction` generally indicates fewer insertions/deletions relative to peers.
- Higher `consensus_identity` indicates higher similarity to the MSA consensus.

### `results/report.csv` column definitions

The final report is built by merging sequence-level features with structure predictions
and then computing a composite developability score.

Core identifiers:

- `rank`: Position after sorting by `developability_score` (1 = best in this run).
- `sequence_id`: FASTA record identifier.

Physicochemical features (from `scripts/features.py`):

- `length`: Number of amino acids in the cleaned sequence.
- `molecular_weight`: Biopython protein molecular weight estimate.
- `hydrophobicity`: Mean Kyte-Doolittle-style residue score over the sequence,
	using the `HYDRO_SCALE` mapping in the code.
- `net_charge`: Approximate net charge proxy computed as:
	- positive = `count(K) + count(R) + 0.1 * count(H)`
	- negative = `count(D) + count(E)`
	- `net_charge = positive - negative`
- `charge_balance`: Charge-neutrality preference term:
	- `charge_balance = 1 / (1 + abs(net_charge))`
	- Values near 1 indicate near-neutral charge.
- `freq_A` ... `freq_Y`: Amino acid composition frequencies for each residue in
	`ACDEFGHIKLMNPQRSTVWY`.

Structure fields (from `scripts/structure.py`):

- `confidence`: Backend-provided confidence proxy.
	- `mock` backend: random uniform in `[0.65, 0.95]`.
	- `esmfold`: derived from mean pLDDT (`plddt_mean / 100`, clipped to `[0, 1]`).
	- `alphafold3` / `rosetta`: uses API-provided confidence when available,
	  otherwise derived from mean PDB B-factor as a proxy.
- `stability`: Backend-provided stability proxy used directly by scoring.
	- `mock` backend: random uniform in `[0.40, 0.95]`.
	- `esmfold` / `alphafold3` / `rosetta`: currently set equal to confidence (pipeline proxy).
- `plddt_mean`: Mean per-atom pLDDT from the returned ESMFold PDB structure (esmfold only).
- `backend`, `backend_requested`, `backend_used`, `notes`, `fallback_reason`:
	provenance/debug columns describing which predictor was requested, actually used,
	and whether fallback occurred.
- `pdb_file`: path to the saved PDB file for that sequence (when available).

Optional MD fields (if `use_gromacs=true` and MD data is present):

- `md_rmsd`: Placeholder RMSD metric.
- `md_stability_adjustment`: Small correction term (placeholder range `[-0.08, 0.08]`).
- `md_backend`: MD backend provenance string.

Scoring intermediates and final score (from `scripts/scoring.py`):

- `hydrophobicity_norm`: Normalized hydrophobicity:
	- `hydrophobicity_norm = clip((hydrophobicity + 4.5) / 9.0, 0, 1)`
- `developability_score` (before optional MD adjustment):
	- `0.4 * stability + 0.3 * hydrophobicity_norm + 0.3 * charge_balance`
- If MD is enabled and available, the score is adjusted to:
	- `clip(developability_score + 0.1 * md_stability_adjustment, 0, 1)`

### What "developability" means in this pipeline

In this repository, `developability_score` is a lightweight composite heuristic,
not a validated experimental developability assay.

Interpretation in this workflow:

- Higher `stability` increases score (largest weight, 0.4).
- More favorable `hydrophobicity_norm` increases score (weight 0.3).
- Near-neutral charge (high `charge_balance`) increases score (weight 0.3).
- Optional MD contributes only a small correction (+/- 0.008 max from the default
	placeholder range and 0.1 multiplier).

Important caveats:

- With `structure_backend: mock`, stability/confidence are randomized placeholders,
	so rankings are illustrative and can vary between runs.
- With `structure_backend: esmfold`, confidence is model-derived, but this workflow still uses
	lightweight scoring assumptions (for example, stability proxy = confidence).
- Therefore, use this score for pipeline demonstration and relative prioritization
	within this toy setup, not as a substitute for wet-lab developability testing.

## ESMFold Notes

- ESMFold backend (`structure_backend=esmfold`) sends each cleaned sequence to the
  configured API endpoint (`esmfold_api_url`) and parses the returned PDB text.
- If the API call fails (network/rate-limit/service error), the workflow logs the
  reason and falls back to the `mock` backend for that sequence.

## AlphaFold3 and Rosetta Notes

- `structure_backend=alphafold3` and `structure_backend=rosetta` are real predictors
	implemented through configurable API calls.
- You need your own running AlphaFold3/Rosetta service instances (typically local, on-prem, or self-hosted) and must point the config URLs to those services.
- This repository does not ship or host AlphaFold3/Rosetta servers for you.
- Set `alphafold3_api_url` / `rosetta_api_url` to your service endpoints.
- Request/response keys are configurable so you can adapt to different API schemas.
- If a backend is not configured or its API call fails, the pipeline logs the reason
	and falls back to `mock` for robustness.

## Notebook Walkthrough

The notebook `notebooks/analysis.ipynb` is an end-to-end exploration of pipeline outputs.
It is designed to be run after a successful Snakemake run.

What it does:

1. Loads and previews `results/report.csv`.
2. Plots the `developability_score` distribution.
3. Shows the top-ranked sequences by `rank` and score.
4. Loads `data/msa.fasta` and visualizes the alignment with:
	- residue-encoded heatmap,
	- gap map,
	- per-position conservation and gap fraction,
	- per-position Shannon entropy,
	- sequence logo-like residue frequency bars,
	- top variable positions table.
5. Loads and displays `data/msa_summary.csv`.
6. Selects the best-developability sequence from `results/report.csv` and renders
	its PDB structure interactively with `py3Dmol` from `data/pdb`:
	- backbone/cartoon in spectrum colors,
	- sidechain spheres (elemental colors) for residues that are at non-100%-conserved MSA positions,
	- printed sequence ID, score, and variable residue labels (3-letter + position).

If MSA is disabled (`run_msa=false`) or PDB files are not available (for example with
`structure_backend=mock`), the notebook shows informative messages instead of failing.

## Notes

- Structure prediction and MD simulation are intentionally lightweight placeholders to keep the pipeline runnable everywhere.
- Local `venv` mode is a lightweight Snakemake workflow; optional GROMACS execution is Docker-only.
- Heavy scientific tools (ESMFold, AlphaFold, Rosetta, GROMACS) can be integrated by replacing stub/placeholder logic while preserving interfaces.
